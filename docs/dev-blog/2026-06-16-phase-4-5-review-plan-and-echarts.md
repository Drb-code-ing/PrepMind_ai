# Phase 4.5.1：把“今天复习”扩展成可预期的学习计划

今天的目标不是继续堆一个新页面，而是补齐 FSRS 复习闭环里一个很关键的体验缺口：用户不只需要知道“今天要复习什么”，也需要提前知道未来几天的复习压力。

## 为什么要做 `/plan`

Phase 4.3 之后，`ReviewTask` 已经能持久化今日任务，Phase 4.4 又补上了离线评分和幂等提交。但这个链路仍然偏“当天执行”：用户只有进入今日任务时，才知道今天有多少卡片到期。

所以 Phase 4.5.1 先做一个轻量的长期计划预览：

- 看未来 7 天每天有多少卡片到期。
- 提示高峰日，避免复习压力突然堆积。
- 只读预览未来压力，不提前创建未来 `ReviewTask`。
- 今日仍然通过 `/today` 执行，未来日期只做计划提醒。

这个边界很重要。如果 `/plan` 提前创建 future task，后续跳过、恢复、离线评分、重复生成都会变复杂。当前阶段只需要稳定回答“未来会发生什么”，不需要把未来任务实体化。

## 后端怎么做

新增接口：

```text
GET /review-tasks/plan?days=7&startDate=2026-06-16&timezoneOffsetMinutes=-480
```

核心数据来源是 `Card.nextReview`。服务端按用户隔离读取复习卡，把 `nextReview` 映射到用户本地日期，再聚合到未来 N 天窗口。

这个接口只做三件事：

1. 统计逾期、今日到期和未来到期卡片。
2. 聚合每日到期数量与预计用时。
3. 读取本地待同步评分数量，用来提示用户仍有离线操作未完成同步。

它明确不做两件事：

- 不调用 `/review-tasks/today` 的懒生成逻辑。
- 不创建未来 `ReviewTask`。

测试里也专门覆盖了这一点：plan 查询可以读 `Card` 和已有 `ReviewTask`，但不能触发 `reviewTask.createMany`。

## 前端怎么做

前端新增 `/plan` 页面，入口放在今日任务之后。这个位置符合用户路径：先从今天开始学习，再向后看未来压力。

页面结构保持轻量：

- 顶部总结未来压力：总量、逾期、今日、未来、待同步。
- 中间用 ECharts 展示未来 7 天柱状趋势。
- 下方按日期列出每日安排。
- 今日行提供“去今日任务”，未来日期只显示“到期后处理”。

另外加了一个本地日期刷新 hook。页面如果跨过午夜、窗口重新获得焦点或从隐藏状态恢复，会重新计算当前日期和查询窗口，避免 PWA 长时间挂后台后计划还停在昨天。

## 为什么升级 `/stats` 到 ECharts

之前 `/stats` 已经能展示数据，但图表表达偏基础。Phase 4.5.1 顺手把它升级成更适合展示的 ECharts：

- 复习趋势：面积折线图，适合看 7 天 / 30 天节奏。
- 评分分布：环形图，看 Again / Hard / Good / Easy 的比例。
- 卡片状态：环形图，看 NEW / LEARNING / REVIEW / RELEARNING 分布。

ECharts 是浏览器图表库，不能直接参与 SSR。为避免 Next.js hydration 问题，项目新增了 `BaseEChart` 客户端组件，只在 `useEffect` 中动态 `import('echarts')`，并在组件卸载时 dispose 实例。

这次还保留了评分分布的文本网格。图表负责视觉表达，文本负责移动端和空数据时的明确读数。

## 一个审查中发现的小问题

最终审查时发现 `/stats` 的空态判断混用了两个概念：

- 当前统计窗口内的复习次数。
- 最近复习日志总数。

如果用户最近 7 天没有复习，但历史日志存在，页面会渲染当前窗口全零的图表，而不是展示当前窗口空态。这会误导用户。

修复后，图表空态只看当前窗口 `stats.totalReviews`；最近记录列表作为独立信息存在，不再影响当前窗口图表是否展示。

## 验收结果

本次验证覆盖了服务端、前端、类型包、数据库包、FSRS 包和浏览器真实交互：

- `/review-tasks/plan` 能返回未来计划，并保持只读。
- `/plan` 有数据和空数据都能展示。
- `/stats` 三个 ECharts canvas 均能正常绘制。
- 7 天 / 30 天切换正常。
- `/today` 入口没有回归。
- 移动端 390px 宽度下 `/plan` 和 `/stats` 无横向溢出。

Phase 4.5.1 完成后，复习系统从“今天执行”向“未来可预期”迈了一步。下一步更适合继续做 Phase 4.5.2：把提醒策略和更长期的计划设置补上，让用户不仅能看到压力，还能被合理地提醒和安排。
